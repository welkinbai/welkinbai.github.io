    一、Mysql支持的数据类型

    Mysql支持的类型大概可分为数字、日期、字符串
    下面分别介绍：

    1.数字

    类型 大小 范围 用途
    TINYINT 1字节 (-128，127) 特小整数
    SMAILINT 2字节 (-2^15,2^15-1) 小整数
    MEDIUMINT 3字节 (-2^23,2^23-1) 中等整数
    INT 4字节 (-2^31,2^31-1) 整数
    BIGINT 8字节 (-2^63,2^63-1) 大整数
    FLOAT 4字节 (-2^128,2^128) 单精度浮点
    DOUBLE 8字节 (-2^1024,2^1024) 双精度浮点
    DECIMAL 不确定 不确定 精确计算

    REAL是DOUBLE的同义词，NUMERIC是DECIMAL的同义词。
    FLOAT和DOUBLE的范围我写的不是特别精确，因为这实际上依赖于Mysql运行的平台是如何实现的。不过它们都拥有一个靠近自然数0的最小值，不能平滑地过渡到无穷小
    另外，这里列出的都是有符号类型的范围。无符号类型的话，就是在前面加上UNSIGNED关键字，范围都会变成正数。
    如何记忆范围里的数字呢？其实不用死记硬背，只需要记得前面占用几个字节即可，1个字节等于8bit，因此就是2的8次方。有符号数的范围由于一个bit用来记录正负，因此只能取到负的2的7次方到2的7次方-1。

    2.字符串

    类型 大小 用途
    CHAR 0-255字节 定长字符串
    VARCHAR 0-65535字节 变长字符串
    TINYBLOB 0-255字节 小二进制字符串
    TINYTEXT 0-255字节 小文本字符串
    BLOB 0-65535字节 二进制字符串
    TEXT 0-65535字节 文本字符串
    MEDIUMBLOB 0-(2^24-1)字节 中等二进制字符串
    MEDIUMTEXT 0-(2^24-1)字节 中等文本字符串
    LONGBLOB 0-(2^32-1)字节 大二进制字符串
    LONGTEXT 0-(2^32-1)字节 大文本字符串

    除此之外，还有BINARY、VARBINARY、ENUM、SET四种类型
    CHAR和VARCHAR:

    • VARCHAR是变长的，即数据多长，就占用多长的存储空间，比较省空间
    • VARCHAR需要额外的1个字节来存储字符串长度，如果字符串长度大于255，则需要2个额外的字节
    • VARCHAR不太适合更新频繁的场景，因为当长度改变后，行的长度就改变了，存储引擎需要做额外的工作，影响性能，产生碎片。
    • Mysql5.0或更高版本，会在存储和检索时保留VARCHAR的空格，CHAR末尾的空格会被自动去除
    • CHAR适合频繁更新，字符串长度不怎么变的场景，比如存放MD5
    • InnoDB这个存储引擎会把过长的VARCHAR转成TEXT
    • 虽然VARCHAR是变长的，但是建表的时候，还是要分配真实需要的空间，不要取得过大，因为这会导致Mysql在使用内存临时表排序或者操作时消耗更多的内存

    BINARY和VARBINARY:

    • 和CHAR、VARCHAR是类似的，不同的是，存储的是二进制字符串，这样存储的是字节码而不是字符，就与字符集无关了
    • 由于存储的是字节码，因此比较起来比字符简单，故查询的性能应该好一些
    • 填充采用的是\0而不是空格(由于我看的书是中文翻译，所以这里我没看懂什么意思……哎，还是原生好啊)

    BLOB和TEXT:

    • 它们都是为了存储很大的数据设计的字符串数据类型(这里我就要吐槽一下既然如此为什么还有TINYBLOB这种东西……)
    • SMALLBLOB=BLOB,SMALLTEXT=TEXT，即它们是同义词
    • 这两种类型当值太大时，Mysql会在行内存储它们的指针，实际内容放在外部单独的区域存储。
    • BLOB和TEXT的区别就是BLOB存储的是二进制的数据，没有字符集也没有排序规则而TEXT有。
    • BLOB和TEXT列进行排序和其他的不同，只会对最前面max_sort_length个字节做排序，用户可以手动设置这个值也可以使用
      ORDER BY SUSTRING(column,length)
      来指定取前多少个字符排序。
    • Mysql没法对BLOB、TEXT列全部长度的字符串进行索引

    ENUM:

    • 枚举类型是特殊的字符串类型，定义枚举列后，真正存在表中的是整数，表的.frm文件则保存整数和枚举字符串的映射关系
    • 如：CREATE TABLE enum_test(e ENUM('fish','apple','dog') NOT NULL);在表中，真正存储的是，1、2、3这样的数字
    • 不要使用数字作为枚举字符串常量，如ENUM(‘1’,’2’,’3’)。这样会导致混乱
    • 枚举的顺序是按照背地里的数字来排序的，因此，你的Order by语句可能得不到按字符串排序的结果。解决方案就是声明的时候就把字符串排好序，枚举常量对应的数字是和声明时的顺序有关的。还有一种方案是使用如下FIELD语句：
    • ENUM和CHAR(VARCHAR)类型关联查询，会慢一些，因此，假如预先知道某列需要与CHAR类型关联，那么就不应该将该列设置为ENUM类型
    • ENUM类型的列可有效缩小表所占的空间，书中写可缩小1/3

    SET:

    • 和ENUM最大的不同就是它的值可以是多个字符串常量的集合，比如设定了字符串常量有：’a’,’b’,’c’。它的某个值可以是’a’,’b’或者’a’,’c’。而ENUM的值只能是它们的其中一个
    • 还有就是SET存储值的方式是使用位，比如abc三个值，可能分别用3个位来存储，如果a和c对应的位为1则说明存储了’a’,’c’

    3.日期和时间

    类型 大小 范围 格式 用途
    DATE 3字节 1000-01-01/9999-12-31 YYYY-MM-DD 日期值
    TIME 3字节 ‘-838:59:59’/‘838:59:59’ HH:MM:SS 时间值或持续时间
    YEAR 1字节 1970/2069(2字符长度格式)或1901/2155(4字符长度格式) YY或YYYY 年份值
    DATETIME 8字节 1000-01-01 00:00:00/9999-12-31 23:59:59 YYYY-MM-DD HH:MM:SS 混合日期和时间值
    TIMESTAMP 4字节 1970-01-01 00:00:00/2037 YYYYMMDDHHMMSS 存放时间戳

    注意：MYSQL 5.5以及以下版本不支持毫秒精度！存储DATETIM和TIMESTAMP时会自动对毫秒进行四舍五入！5.6版本支持，但是要配置打开，并在相应字段声明时，声明精度。如：TIMESTAMP(3)，最高精度可到6。

    注：Mysql的行具有大小限制，一行最大长度为65,535 bytes。因此，假定所有的列都是CHAR(255),字符编码UTF-8，那么表格最大的列数就是65535/(255*3)=85详情请看官网文档：
    Mysql-Limits on Table Column Count and Row Size

    四、一些常用管理命令

    注意：必须以分号结尾

    1. USE schema名字
      schema在Mysql中等同于数据库的概念(下文全部称数据库)。使用该命令以后，你接下来所有的命令都默认是对该数据库操作的。

    2. SHOW DATABASES
      可以列出所有数据库

    3. SHOW TABLES
      在你使用use语句后，列出你指定的数据库的所有表

    4. SHOW COLUMNS FROM 数据表
      显示某个表所有的列信息

    5. SHOW INDEX FROM 数据表
      显示某个表所有的索引

    6. SHOW TABLE STATUS LIKE [FROM db_name] [LIKE ‘pattern’] \G
      显示某数据库下所有表或名字满足pattern的表的各类信息

    7. CREATE {DATABASE | SCHEMA} [IF NOT EXISTS] db_name [create_specification]
      创建一个数据库。[]括起来的是可选值
      IF NOT EXISTS 可在数据库已经存在的时候忽略执行这句话从而避免报错
      create_specification:
      [DEFAULT] CHARACTER SET [=] charset_name
      | [DEFAULT] COLLATE [=] collation_name
      CHARACTER SET指定数据库的默认字符集，
      COLLATE指定数据库对应字符集的字符校对规则(影响排序和字符比较)

    8. DROP {DATABASE | SCHEMA} [IF EXISTS] db_name
      删除数据库。注意：删除数据库操作会导致数据永久删除。另外，如果user表中定义了某用户对你删除的表的种种权限，这些记录不会跟着删除。

    五、一些高级命令(比较复杂的)

    1)创建表

    CREATE [TEMPORARY] TABLE [IF NOT EXISTS] tbl_name
    (create_definition,…)
    [table_options]
    [partition_options]

    []括起来的是可选值。

    • TEMPORARY：这个字段允许你创建一个临时表，当会话(session)结束时，这个临时表自动销毁。临时表赋予你不同会话创建同名表的能力，甚至是和已有的非临时表同名。
      注意：临时表的创建不会自动commit事务，临时表和数据库关系是很松的，删除数据库不会删除其中的临时表，同样地，你还可以定义临时表在压根不存在的数据库里，不过之后的每次调用你都要用数据库名.临时表名的方式访问

    • create_definition:
      这部分内容主要有以下结构组成：

      • col_name:就是列名
      • column_definition:
        又由以下结构构成：
        data_type [NOT NULL | NULL] [DEFAULT default_value]
        [AUTO_INCREMENT] [UNIQUE [KEY] | [PRIMARY] KEY]
        [COMMENT ‘string’]
        [COLUMN_FORMAT {FIXED|DYNAMIC|DEFAULT}]
        [STORAGE {DISK|MEMORY|DEFAULT}]
        [reference_definition]
        • data_type：就是column的类型
        • NOT NULL或NULL：指定列是否可以为空
        • DEFAULT:指定默认的值，必须为常数，不能是函数，不能为BLOB和TEXT类型设置默认值，自增的列也不行
        • AUTO_INCREMENT:设置列为自增列，只能设置整数(integer)和单精度浮点(floating-point)类型为自增，超过最大值时，自增列又会从1开始。自增数字都是正的，不能是负的
        • UNIQUE：唯一性索引，设置列里的值全部是唯一的，不允许插入已经有的值。另外，NULL是可以插入多个的
        • PRIMARY：主键，不用说了，必须非空，一种唯一性索引，因此你不能再创建一个索引名字叫PRIMARY。在InnoDB表中要尽可能地使主键更短，来优化secondary index(辅助索引)占用的空间。最好先声明PRIMARY列再次是UNIQUE列，最后是其他列，这样可以提高索引使用效率
        • COMMENT：列的注释，1024个字符长度限制
        • COLUMN_FORMAT：针对NDB引擎的字段，指定列是动态的还是固定的空间存储还是根据类型自动选择。其他引擎声明了也没用。
        • STORAGE：针对NDB引擎的，指定列是存放内存里还是硬盘上。其他引擎没用。必须与表空间声明字段TABLESPACE配合使用。
        • reference_definition:
          主要是声明外键约束用的，InnoDB和NDB引擎支持。由以下结构组成：
          REFERENCES tbl_name (index_col_name,…)
          [MATCH FULL | MATCH PARTIAL | MATCH SIMPLE]
          [ON DELETE reference_option]
          [ON UPDATE reference_option]
          • tbl_name就是其他表的表名
          • index_col_name就是其他表已经建立过索引的列的列名。就是说其他表的列最好是非空并唯一的。
          • MATCH语句应该避免使用，尤其是InnoDB引擎，默认就是MATCH SIMPLE。如果显示声明了，可能导致其连同后面的语句一起被忽略。这个语句主要是指定外键是否允许或部分允许空值NULL的
          • ON DELETE：删除时候的行为，有四种选项：RESTRICT(有外键不给删) | CASCADE(递归删) | SET NULL(置为空) | NO ACTION(不做事情)。
          • ON UPDATE：更新时候的行为，有四种选项：RESTRICT(有外键不给更新) | CASCADE(递归更新) | SET NULL(置为空) | NO ACTION(不做事情)。
      • CONSTRAINT就是约束的意思，所以接下来可选的句子都是设置表的约束条件的。大部分都很容易看懂字面意思，就不详细说了。
      • index_option：
        索引的配置项，其中PARSER是为全文索引用的，全文索引需要特殊的解析插件的话可以定义。
        index_type有BTREE和HASH两种。不解释了。
        KEY_BLOCK_SIZE有点儿歧义，官网文档针对不同的引擎有不同的解释，针对MyISAM引擎，可以用来指定索引块的大小。针对InnoDB引擎，可以用来指定压缩过的表的分页大小(不能超过innodb_page_size)。无论哪种引擎，这个值都只是一个参考值，真正运行中引擎会根据情况调整。
        index_option:
        KEY_BLOCK_SIZE [=] value
        | index_type
        | WITH PARSER parser_name
        | COMMENT ‘string’

    • table_options：
      指定的是表级别的设置，有如下选项：
      table_option:
      ENGINE [=] engine_name
      | AUTO_INCREMENT [=] value
      | AVG_ROW_LENGTH [=] value
      | [DEFAULT] CHARACTER SET [=] charset_name
      | CHECKSUM [=] {0 | 1}
      | [DEFAULT] COLLATE [=] collation_name
      | COMMENT [=] ‘string’
      | CONNECTION [=] ‘connect_string’
      | DATA DIRECTORY [=] ‘absolute path to directory’
      | DELAY_KEY_WRITE [=] {0 | 1}
      | INDEX DIRECTORY [=] ‘absolute path to directory’
      | INSERT_METHOD [=] { NO | FIRST | LAST }
      | KEY_BLOCK_SIZE [=] value
      | MAX_ROWS [=] value
      | MIN_ROWS [=] value
      | PACK_KEYS [=] {0 | 1 | DEFAULT}
      | PASSWORD [=] ‘string’
      | ROW_FORMAT [=] {DEFAULT|DYNAMIC|FIXED|COMPRESSED|REDUNDANT|COMPACT}
      | STATS_AUTO_RECALC [=] {DEFAULT|0|1}
      | STATS_PERSISTENT [=] {DEFAULT|0|1}
      | STATS_SAMPLE_PAGES [=] value
      | TABLESPACE tablespace_name [STORAGE {DISK|MEMORY|DEFAULT}]
      | UNION [=] (tbl_name[,tbl_name]…)

      • ENGINE：引擎，不用说了，一般都是InnoDB，还有略旧一点儿引擎MyISAM。引擎介绍可以看我这篇文章： 《高性能Mysql》读书笔记——第一章Mysql架构和历史
      • AUTO_INCREMENT: 指定自增列从什么数字开始自增
      • AVG_ROW_LENGTH：这个官网写的啥啊，反正就是MyISAM引擎用的，用来指定行的平均长度，用来被计算表的数据和索引至多占用多少空间。默认是占256TB，好像如果设置到7，你就可以搞成65,536TB。文末有官网文档地址，谁英语好帮我看看啥意思……
      • CHARACTER SET:字符集设置
      • COLLATE：字符集校对设置
      • CHECKSUM：校验和。仅用于MyISAM。设置成1使得表格维护一个校验和，每次更新校验和也会更新。会拖慢速度，但是更容易通过CHECKSUM TABLE语句找到坏数据的表。
      • COMMENT：表格的注释，没啥好说的，最大2048个字符
      • CONNECTION：FEDERATED引擎，一种可以将表定义放在本地，数据放在远程的表格。其中，远程服务器的连接信息放在这个字段里。
      • DATA DIRECTORY：指定一个特殊的路径来放表的文件。
      • INDEX DIRECTORY：指定一个特殊的路径来放索引的文件。
      • DELAY_KEY_WRITE：设置为1的话，直到表格关闭才会更新key值。仅MyISAM引擎。
      • INSERT_METHOD：为MERGE引擎设计的，这个引擎可以支持将MyISAM引擎的同结构表合并成一个表，便于查询。但是插入的时候就麻烦，不知道要插入到哪个表里去。这个字段定义了插入到哪儿。是合并的第一张表，还是最后一张，还是不让插入。
      • MAX_ROWS：一个参考值，告诉引擎至少要支持到存这么多行。
      • MIN_ROWS：一个参考值，只有MEMORY引擎用，告诉引擎最少打算存多少行。
      • PACK_KEYS：仅MyISAM，压缩索引的。若为1，则压缩所有的。会导致更新更慢，读取更快。0则不压缩，DEFAULT就是仅压缩CHAR,VARCHAR,BINARY,VARBINARY这些类型的列。默认不写的话，只压缩字符串，不压缩数字类型
      • PASSWORD：官方文档说没用了，看来Mysql也会因为历史原因生产死亡的代码……
      • ROW_FORMAT：定义行存储的物理格式。不同引擎不一样，主要就是定义是紧凑还是过去的宽松存储格式，还有压缩神马的，有兴趣的可以看文末官网文档。
      • STATS_AUTO_RECALC：是否自动重新计算表的持久化统计数据。
      • STATS_PERSISTENT：是否开启持久化表的统计数据功能
      • STATS_SAMPLE_PAGES：用于估计索引列的统计数据的基数的时候，设定取样的索引页数目，详情见官网文档
      • TABLESPACE：仅用于NDB引擎。分配表空间的。
      • UNION：就是MERGE引擎用的，来把两个MyISAM同结构表合并
    • partition_options：
      用来控制表的分割方式的，很少用到，还是不讲了，见官网。

    3)查询语句

    • ALL/DISTINCT/DISTINCTROW：ALL其实可以忽略不写，默认就是ALL，DISTINCT和DISTINCTROW是同义词，代表输出结果去重
    • HIGH_PRIORITY：用在MyISAM, MEMORY, and MERGE这类表锁引擎上，代表高优先级，先于更新的语句执行，并锁住表
    • STRAIGHT_JOIN：优化表关联查询的，如果优化器没有以最优的方式关联表，你可以使用这个字段。(强迫优化器按照你声明的顺序关联查询)
    • SQL_BIG_RESULT/SQL_SMALL_RESULT：告诉优化器查询结果是非常大还是很小，以利于优化器优化GROUP BY或DISTINCT的性能。
    • SQL_BUFFER_RESULT：强制性将查询结果放入临时表。帮助MYSQL提前释放表锁，否则还要等到将结果发给客户端后才能释放。只能用在顶级查询，子查询和联合查询(UNION)不行
    • SQL_CACHE/SQL_NO_CACHE：告诉MYSQL要不要缓存查询结果，缓存的话下一次直接从缓存里取结果
    • SQL_CALC_FOUND_ROWS：使用了这个参数的haunted，可以在之后使用SELECT FOUND_ROWS()语句得到结果集有多少行，忽视LIMIT限制。
    • LIMIT：就是制定返回的数据集，offset代表偏移量，比如5的话，就是从第6条数据返回，row_count就是返回的行数。也可以使用LIMIT row_count OFFSET offest这种格式来声明
    • INTO OUTFILE：是用来将结果输出到某个文件或者某个变量里的。
    • FOR UPDATE：会在相应的行上上写锁(InnoDB这种支持的引擎才行)，直到事务结束
    • LOCK IN SHARE MODE：在相应行上加共享锁。详情见另一篇文章：《高性能Mysql》读书笔记——第一章Mysql架构和历史
    • PROCEDURE：存储过程，一段可以处理查询结果的程序
    • PARTITION：和分页有关，5.6.2版本后可以支持分页，将数据自动地按照某些条件分成几页，查询的时候使用这个条件来查询相应页的数据

    有关查询语句的参数就介绍这些。具体的查询操作见下章内容。

    • 多个语句搜到的结果，它们应该拥有相同的结构，如相同的列，列的类型也相同
    • 最终结果集的列名会以第一个select语句的列名为准
    • 只有最后一个select语句允许使用INTO OUTFILE语法来将结果输出到文件。结果输出的是所有语句合并的最终结果。
    • HIGH_PRIORITY关键字不能用在UNION中，如果你声明在第一个select语句中，会被忽略，声明在后面的语句中，会报错。
    • UNION默认的行为是最终结果会进行去重，你也可以使用ALL关键字来不去重。
    • 你可以在UNION语句中即使用ALL又使用DISTINCT，但是DISTINCT语句会将左边所有的ALL语句覆盖掉，这样的话，得到的就是去重的了。
    • 你可以在每个select语句中使用ORDER BY语句，但是必须搭配LIMIT语句，否则没有任何效果，最终结果仍然是没排序的。如果想要最终结果排序，那么就把select语句用括号括起来，在整个语句的最后用ORDER BY来排序。如果有字段起了别名，那ORDER BY 必须用别名来指定列。

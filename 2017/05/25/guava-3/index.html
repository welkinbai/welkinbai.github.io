适用范围

缓存这个不用说了，很多情况下我们的确需要缓存。
Guava文档说了，它提供的缓存比较像ConcurrentMap，但是也是有区别的，我们从上面的例子就能看出来区别，Guava提供的可以设定缓存的过期时间，时间一到entry就被自动移除了，还允许你注册监听器Listener，移除的时候告诉你一声，做点儿额外的逻辑。
总之，总的来说，下面这些情况同时满足时，Guava提供的缓存比较适合你：

  • 你想要花一些内存来提高程序的速度
  • 你预期你的key值会不止一次地被查询
  • 你想要缓存的东西不会撑爆虚拟机的最大内存
  • Guava的缓存是单机缓存，如果你有多台机器干同样的事儿，那么你可能用不上这个缓存，还是考虑Memcached或者redis吧

如果上面的这些都满足了，那么恭喜你，可以使用Guava的缓存了。

基于时间的缓存释放

Guava提供了两个方法：

  • expireAfterAccess(long, TimeUnit)：在上一次读过或者写过，反正就是访问过后，再过一定的时间就释放
  • expireAfterWrite(long, TimeUnit)：在创建好之后，或者上一次值更换之后，再过一定的时间就释放

还可以用CacheBuilder.ticker(Ticker)来指定外部的时钟源

基于引用的缓存释放

有三个方法：

  • CacheBuilder.weakKeys()：key值使用弱引用来保存，即当key值不再有强引用和软引用时，垃圾回收器将回收缓存
  • CacheBuilder.weakValues()：value值使用弱引用来保存，当value值不再有强引用和软引用时，垃圾回收期将回收缓存
  • CacheBuilder.softValues()：value值使用软引用来保存，软引用只有在内存不够用的时候，垃圾回收器才会去回收。并遵循最近最频繁使用的方式（最近最频繁使用的留下？原文没说）。

注意，上面提到的三种方法都会令相应的字段在比较的时候，使用==操作符而不是equals()方法，这是因为垃圾回收器在乎的是引用值，所以比较自然而然也要换成引用值来比较。

明确移除

任何时候，你都可以使用明确的方法来移除缓存而不是消极地等待缓存自己释放：

  • Cache.invalidate(key)：移除某一个缓存
  • Cache.invalidateAll(keys)：移除一批
  • Cache.invalidateAll()：移除全部

移除什么时候会发生？

尽管上面介绍了一堆可以自动释放缓存的方法，实际上，Guava并没有按你想象的那样去维护这个东西。比如基于时间的缓存释放，过期了以后，我们放在缓存里面的东西会立马被移除吗？
答案是，不会。
维护缓存的操作见缝插针地被安排在写操作中，针对写操作比较少的情况，则被安排在读操作中。
也就是说，过期的entry会在下一次写操作或者读操作的时候，被Guava给移除掉。
之所以采取这种做法，显然是实现上面的考虑，如果Guava被设计成在缓存过期的时候立即处理，那么必然要开启一个线程来时刻维护缓存，类似守护进程那个样子。Guava是不希望这么做的，因为这样一方面涉及多线程，会不可避免地与用户的操作产生互斥和锁竞争，增加复杂度，另一方面，有些环境严格控制程序能够创建的线程数，可能导致缓存的守护线程创建失败。
如果你非常在乎缓存的读取性能，怕Guava在读取操作上干别的事儿影响读取速度，你可以自己实现一个守护线程，并调用Cache.cleanUp()方法来通知缓存打扫掉过期的数据。

特性

统计数据

使用CacheBuilder.recordStats()可以打开统计功能，然后使用Cache.stats()就可以得到一个包含统计数据的对象。比如命中率啊、平均读取时间啊、缓存释放次数啊等等和性能有关的值。

asMap

使用asMap可以得到一个视图。有以下要点需要注意：

  • cache.asMap()包含所有此刻在缓存中的entry。
  • asMap().get(key)相当于cache.getIfPresent(key)，但是不会触发缓存写入。
  • 访问时间会被任何写操作和读操作重置，包括cache.asMap().get(Object)和cache.asMap().put(K,V)。但是其他方法，比如遍历啊，是否包含啊之类的方法不会重置访问时间(access time，猜测用在上面的基于时间的缓存释放功能里)。

中断和Bug

似乎Guava的程序员没能很好地处理InterruptedException，给出的理由是大多数缓存都是单线程的，即使多线程中断的场景也很少见，所以让所有人都去在乎InterruptedException不太好。
但是没能很好地处理InterruptedException有可能导致线程的永久阻塞，代码修改是没戏了，因为改起来太难了，于是谷歌的程序员们又新给了一个缓存的实现，AsyncLoadingCache，这个可以很好地处理中断。
这大段的英文翻译起来真心心塞啊，总之就是在多线程环境下有一些bug。

第八条 覆盖equals时需要遵守的约定

什么时候应该覆盖equals?

equals方法默认的逻辑是对比两个变量所指向的是否是同一个对象（实例）。
但是如果你想对比的是逻辑相等呢？比如两个对象都含有value这个域，你只是希望根据value这个域的情况来比较对象呢？
这种对象的类被称为”值类“，比如，代表日期的类Date。
需要注意，枚举类不需要覆盖equals方法，因为枚举类每个枚举值都被限定为全局唯一的实例，因此虽然equals比较的是对象是否相同，但是对象相同保证了其所代表的值一定相同。

覆盖equals必须遵守的约定

首先，覆盖equals方法必须实现如下等价关系：

  • 自反性： 当x不等于null时，x.equals(x)必须等于true
  • 对称性： 当x、y不等于null时，x.equals(y)和y.equals(x)的结果必须相同
  • 传递性： 当x、y、z不等于null时，x.equals(y)和y.equals(z)都返回true，那么x.equals(z)也应该返回true
  • 一致性： 当x、y不等于null时，只要x,y中的相关信息没有被改变，那么无论调用多少次x.equals(y)都应该返回稳定一致的结果，如果是true就永远是true
  • null处理：任何对象调用equals方法和null比较都应该返回false。即x.equals(null)永远为false

如果你要覆盖equals方法，必须满足上述要求，不能偷懒，覆盖好后必须用单元测试测试上述等价关系。
书中列举除了违反上述等价关系会发生什么严重的后果，我就不一一说明了，总之就是一句话，如果不满足上述关系，你的程序可能不会像你想象的那样运行。

覆盖equals时建议的步骤

覆盖equals有一些有用的技巧，建议按照下述步骤进行：

  1. 优先使用==操作符检查入参是否是方法调用者的一个引用，如果是立即返回true，这是一种快速返回策略，如果后续的逻辑非常复杂而且可能会耗费性能，那么这是有必要的
  2. 使用instanceof操作符来对入参的类型做检查，如果不是方法调用者的类型或者不是类所实现的接口，那么立即返回false
  3. 在上面第2步的基础上，就可以放心地将入参转换成正确的类型，不用担心抛出ClassCastException
  4. 对于该类中的每个关键域，比较入参中的值和调用方法的对象中的值是否相等。
    除了float和double的基本类型使用==来比较
    对于float要使用Float.compare，double则使用Double.compare方法，这样特殊处理是因为有类似Float.NaN，-0.0f等特殊的常量。
    对象则使用对象的equals方法来比较
    如果是数组则数组中的元素要按照上述方法递归地处理
    注意有些域为null是合法的，这个时候需要考虑null的情况下比较，使用三目运算符是常用的做法：field==o.field || (field != null && field.equals(o.field))
    注意优先比较可能最可能不相等的域，以快速返回获得更好的性能
    注意不用比较所有的域，如果有些域的值是完全由其他域得出的，那么它就是冗余的域，没必要比较这些域
  5. 你编写完成了，那么你要检查一下，你写的是否满足上面提到的所有等价关系，最好写单元测试来验证它们

额外的一些告诫

  1. 覆盖equals方法时必须同时覆盖hashCode方法，原因见第九条
  2. 不要企图让equals方法过于智能，否则会很容易不满足等价关系或者隐藏深不见底的bug
  3. 不要将覆盖方法的入参由Object改为其他类型，修改参数签名后你写的方法就不再是覆盖父类的方法而是重载方法，那没什么作用。

第九条 覆盖equals时总要覆盖hashCode方法

为什么？

上面提到过这个问题，你也许会问为什么？
最好的回答是如果你只覆盖equals方法而不覆盖hashCode方法，将导致你修改的类不能够在依赖hash散列的类中正常工作，比如HashMap、HashSet、HashTable、HashXXX等。
到时候你会发现对象总是在HashMap里面神秘失踪，或者神秘出现。

覆盖hashCode应该遵守的约定

  1. 在应用程序执行期间，只要对象的equals方法依赖的相关信息没有改变，那么无论调用多少次，hashCode方法针对同一个对象应该始终返回同一个整数。当然应用程序的多次执行之间，可以返回不同的整数。
  2. 如果两个对象由equals方法比较是相等的，那么由hashCode产生的整数就应该相等
  3. 如果两个对象由equals方法比较是不相等的，那么由hashCode产生的整数可以相等也可以不相等，但是程序员应该尽量让其不相等以提升散列表的性能

为什么有这样的约定？

hashCode返回的值主要是用在散列表中，而散列表的工作原理就是根据hashCode的返回结果将对象放在相应的位置，下次要从散列表中取出时，只要使用相同的hashCode整数值，就一定能取出相同的对象。
基于这样的工作原理，就必须要求hashCode应该满足对于同一个对象，始终返回同一个整数。
至于为什么和equals方法有关，那是因为有些散列表的实现仅仅凭借hashCode还不能确定对象的位置，仅仅凭借hashCode确定的位置里可能有多个对象，此时就需要使用equals方法来比较并得到最终的对象。HashMap的实现正是如此，先根据hashCode来得到key，value对在那个hash桶中，然后再遍历桶中的链表来获得key，value对的确切位置（使用equals方法）。

如何产生满足条件的hashCode?

这是一个非常麻烦的技术活儿，就像书中所说，得到一个分布均匀的hash函数是非常困难的，恐怕要留给科学家们去完成，如果我们想要性能和结果非常优秀的hashCode方法，恐怕要借助于一些第三方的写好的工具库，JAVA的世界最令人赞叹的就是，当你需要什么，有人一定已经实现过了。
如果不想要这么麻烦，书中给出了一个简单的方法生成还算可以的hashCode：

  1. 保存一个非零的常数值比如17到名为result的int变量中
  2. 为对象中的每个关键域（就是除了冗余域的域，equals方法使用到的域）计算散列码c：
    如果是boolean类型，则计算f?1:0
    如果是byte、char、short、int类型，则计算(int)f
    如果是long类型，则计算(int)(f^(f>>>32))
    如果是float类型，则计算Float.floatToIntBits(f)
    如果是double类型，则计算Double.doubleToLongBits(f)得到long类型，然后在根据long类型得到结果
    如果是对象，就看equals方法中比较这个对象时是否调用了这个对象的equals方法，如果调用了，那么这里也调用hashCode方法来得到结果，如果不是，则需要为这个域计算一个范式（我理解就是再为其实现一个类似hashCode的逻辑，将其标准化为一个数值，比如使用JSON，最终统一会得到一个string，当然JSON代价太高，只是举例），然后对范式得到的结果来调用hashCode。如果对象是null，则返回0
    如果域是数组，那么数组中的每个元素都要递归地应用上面的规则，然后按照第3步的方法把结果组合起来。
  3. 使用下面的公式将上面得到的结果都加进result里面：
    result = 31 * result + c
  4. 返回result

好了，现在你要做的就是写单元测试，验证你的hashCode方法是不是满足上面的约定

一些解释

1.为什么result要初始化为17或者其他常量？
答：因为假如没有这个设定，而我们的对象的很多域的初始化值都为0的话，会增加哈希冲突的可能性。如果有了一个常量，则对象得到的hashCode不会完全取决于其域值，有助于减少hashCode。
2.为什么要乘以31？
答：那是因为31是一个奇素数，偶数总会有一个约数为2，而2的乘法在计算机中等同于移位运算，这种移位运算可能导致高位溢出，丢失信息。奇素数降低这种可能性。
另外，31参与乘法运算时，在现代VM中可以被自动优化，即 31*i == (i<<5)-i

额外的一些告诫

  1. 绝对不要为了提高性能，只使用对象的一部分关键域来做hashCode，也许去掉一个域算出来的hashCode仍然是满足条件的，但是当有成千上万个对象产生的时候，这种细微的差别被无限放大，最终导致还是会有不同的对象得到同样的hashCode，而你使用单元测试的时候注意不到这些。
  2. 绝对不要依赖hashCode返回的值来做额外的逻辑，因为一旦做了那相当于定死了hashCode方法的实现逻辑，以后如果想要使用更好的散列函数去替代实现，就会发现那些依赖返回值所做的特殊逻辑可能产生问题。

第十条 始终覆盖toString

这一条没什么好说的，始终覆盖toString可以在日志输出的时候带来方便。何况，现代IDE已经可以自动地做这种事，只需要你轻轻一点，toString函数就自动生成好了，还有多种风格可选。

第十一条 谨慎地覆盖clone

Object类中有个clone方法，注意，该方法的默认实现是一个浅拷贝。
我们还有一个接口叫Cloneable，这个接口什么方法都没有，是个标识接口，如果你要对某个类的对象调用clone方法，这个类必须实现cloneable接口。
如果覆盖clone方法，就要注意类的内部是否拥有可变的对象引用或者存储可变对象引用的数组或哈希表，如果有这些东西，那么你必须想办法递归地去将内部的引用对象也拷贝一份，否则你覆盖的实现也是一个浅拷贝。浅拷贝产生的对象在这种情况下会有各种诡异的行为，因为它内部的域指针和另一个对象内部的域指向同一个对象。
像书中说的，覆盖clone方法并实现Cloneable接口是如此地复杂并且容易出错，与之相比，提供一个静态工厂方法，来专门拷贝对象（深拷贝对象），来的更加高效和实际一些。毕竟如第一条所说，静态工厂方法非常灵活好用。
所以结论就是，设计良好的接口或者类，都绝不会去实现Cloneable接口，也不会轻易覆盖clone方法，有些专家级的程序员干脆从来不覆盖clone方法。

第十二条 考虑实现Comparable接口

书中认为，所有的值类都应该实现Comparable接口，很遗憾很少有人这么做。
如果你实现Comparable接口，最好满足下面几条约定(假设返回值都是-1,0,1三个数中的一个)：

  • x.compareTo(y)==-y.compareTo(x)
  • x.compareTo(y)>0且y.compareTo(z)>0，则x.compareTo(z)>0 即传递性
  • x.compareTo(y)==0时，x.compareTo(z)==y.compareTo(z)
    可选地，还需满足下面的约定：
  • x.compareTo(y)==0 的结果和x.equals(y)的结果相同

很少有人在实现的时候考虑上面全部的约定，这在绝大多数情况下都没有问题，但是一旦发生问题，我敢保证那将是一个令人头痛，浪费大量时间仍然毫无头绪的bug。
很多情况下，当程序员需要使用到和顺序有关的接口时，会手动地去实现一个匿名内部类Comparator，这也是可以的，但也许实现Comparable接口的方法更加通用一些，也减少了匿名内部类的写法。
注意无论哪种实现方法都要遵循上述约定，另外，比较值类的域时，比较顺序很重要，优先比较那些最可能不相同的关键域可以提高性能。
compareTo方法其实没有规定返回值一定是-1,0,1，只要符号对了就行，因此有时候你可以直接使用减法来简化代码（只是要注意两个数相减得到的值不能比INTEGER.MAX_VALUE大）。
